## 📋 实体关系详细说明

### 1. 工资表 (salary_table) ↔ 汇总表 (summary_table)

**关系类型**：1:1映射  
**关联字段**：姓名  
**业务含义**：
- 工资表存储原始工资数据
- 汇总表在工资表基础上增加状态字段
- 字段1:1精准映射，确保数据一致性

**映射规则**：
```
salary_table.序号 → summary_table.序号
salary_table.部门 → summary_table.部门
salary_table.姓名 → summary_table.姓名
salary_table.入职日期 → summary_table.入职日期
... (共19个字段1:1映射)
```

---

### 2. 用户表 (users) ↔ 签名文件表 (signature_files)

**关系类型**：1:N  
**关联字段**：姓名  
**业务含义**：
- 一个用户可以上传多个签名文件
- 每个签名文件属于一个用户
- 支持签名文件版本管理

**外键约束**：
```sql
signature_files.用户姓名 → users.姓名 (ON DELETE CASCADE)
```

---

### 3. 用户表 (users) ↔ 操作日志表 (operation_logs)

**关系类型**：1:N  
**关联字段**：姓名  
**业务含义**：
- 一个用户可以产生多条操作日志
- 每条操作日志记录一个用户的操作行为
- 支持审计追溯

**外键约束**：
```sql
operation_logs.操作人 → users.姓名 (ON DELETE CASCADE)
```

---

### 4. 汇总表 (summary_table) ↔ 签名文件表 (signature_files)

**关系类型**：N:1  
**关联字段**：签名图片 → 文件ID  
**业务含义**：
- 汇总表记录最新签名文件ID
- 通过文件ID关联到具体签名文件
- 支持签名文件历史查询

**关联方式**：
```
summary_table.签名图片 = signature_files.文件ID
```

## 🔄 数据流转关系

### 用户登录流程

```
1. 用户输入姓名
   ↓
2. 查询 users 表验证身份
   ↓
3. 查询 salary_table 获取工资信息
   ↓
4. 生成 Token 并更新 users 表
   ↓
5. 记录操作日志到 operation_logs 表
```

### 工资查询流程

```
1. 用户携带 Token 请求
   ↓
2. 验证 Token 有效性
   ↓
3. 从 salary_table 查询工资数据
   ↓
4. 更新 summary_table 查询状态和次数
   ↓
5. 记录操作日志到 operation_logs 表
```

### 签名上传流程

```
1. 用户上传签名文件
   ↓
2. 验证文件格式和大小
   ↓
3. 生成文件ID并存储文件
   ↓
4. 插入记录到 signature_files 表
   ↓
5. 更新 summary_table 签收状态
   ↓
6. 从 salary_table 复制数据到 summary_table
   ↓
7. 记录操作日志到 operation_logs 表
```

## 🎯 核心业务场景

### 场景1：用户登录并查看工资

**涉及表**：users, salary_table, summary_table, operation_logs

**执行步骤**：
1. 查询 users 表验证姓名
2. 查询 salary_table 获取工资数据
3. 更新 summary_table 查询状态
4. 插入 operation_logs 记录登录操作

### 场景2：用户上传签名文件

**涉及表**：signature_files, summary_table, salary_table, operation_logs

**执行步骤**：
1. 插入 signature_files 记录文件信息
2. 从 salary_table 复制数据到 summary_table
3. 更新 summary_table 签收状态
4. 插入 operation_logs 记录上传操作

### 场景3：管理员查看汇总表

**涉及表**：summary_table, signature_files, operation_logs

**执行步骤**：
1. 查询 summary_table 获取汇总数据
2. 关联 signature_files 获取签名文件信息
3. 插入 operation_logs 记录查看操作

## 🔒 数据完整性保证

### 实体完整性
- 所有表都有主键约束
- 主键自动递增，确保唯一性

### 参照完整性
- 外键约束确保关联数据一致性
- 级联删除确保数据清理

### 域完整性
- 数据类型约束确保数据格式正确
- CHECK约束确保数据范围合理
- NOT NULL约束确保必填字段

### 用户自定义完整性
- UNIQUE约束确保唯一性
- 业务规则约束（如实际出勤天数≤应出勤天数）

## 📈 性能优化建议

### 查询优化
1. 合理使用索引，避免全表扫描
2. 使用视图简化复杂查询
3. 避免在WHERE子句中对字段进行函数操作

### 存储优化
1. 定期清理历史日志数据（保留≥6个月）
2. 定期备份数据库
3. 对大表进行分区管理

### 并发优化
1. 使用事务确保数据一致性
2. 合理设置锁级别
3. 避免长事务
